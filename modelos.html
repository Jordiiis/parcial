Introducción Modelos Casos de Uso Recursos

Los 4 Pilares de NoSQL: Diversidad y Propósito

Cada modelo de datos NoSQL está diseñado para resolver un tipo específico de desafío, optimizando el rendimiento y la escalabilidad.

1. Bases de Datos Documentales

Concepto: Almacenan datos en estructuras de documentos semiestructurados, comúnmente en formato **JSON** o BSON. Cada documento es auto-contenido y su estructura puede variar.

Fortaleza: Flexibilidad total del esquema. Son intuitivas para los desarrolladores porque el modelo de datos se mapea fácilmente a objetos de programación (POJOs, POPOs).

Uso Ideal: Catálogos de productos, gestión de contenido (CMS), perfiles de usuario.

2. Bases de Datos Clave-Valor

Concepto: El modelo más simple. Almacena datos como un gran diccionario o hash map, donde una clave única apunta a un valor (que puede ser simple o complejo).

Fortaleza: Velocidad y latencia extremadamente bajas (O(1)). Ideal para operaciones de lectura/escritura rápidas y simples.

Uso Ideal: Caching de alto rendimiento, gestión de sesiones de usuario, carritos de compra.

3. Bases de Datos de Columna Ancha

Concepto: Almacenan datos en "familias de columnas" en lugar de filas. Permiten una gran distribución y eficiencia para almacenar datos dispersos.

Fortaleza: Escalabilidad masiva para operaciones de escritura intensiva (Big Data). Alta disponibilidad y tolerancia a fallos en clústeres distribuidos.

Uso Ideal: Almacenamiento de logs, telemetría (IoT), series temporales.

4. Bases de Datos de Grafo

Concepto: Almacenan datos como nodos (entidades) y bordes (relaciones). El foco principal es la conexión entre los datos.

Fortaleza: Consultas de relaciones complejas (ej: "amigo de un amigo") que son extremadamente lentas o imposibles en SQL.

Uso Ideal: Redes sociales, sistemas de recomendación, detección de fraude.

Comparativa: SQL vs. NoSQL

Característica SQL (Relacional) NoSQL (No Relacional)
Modelo de Datos Tablas fijas (filas y columnas). Flexible (Documento, Clave-Valor, Grafo, etc.).
Esquema Rígido (se define antes de escribir datos). Dinámico (se define en la aplicación).
Escalabilidad Vertical (aumentar potencia del servidor). Horizontal (distribuir en múltiples servidores).
Garantías (Teorema) ACID (Consistencia fuerte). BASE (Disponibilidad y consistencia eventual).
Mejor para... Transacciones financieras, datos estructurados. Big Data, desarrollo ágil, datos no estructurados.
